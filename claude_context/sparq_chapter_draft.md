# SPARQ: System for Pathogen Analysis, Reasoning and Querying

## 1. Motivation

The bottleneck in *Salmonella* surveillance analysis is not statistical method but the orchestration of analysis itself — and this orchestration can be delegated to a structured agentic system without sacrificing reproducibility. The prevailing alternative makes this design claim legible by contrast. Contemporary surveillance analysis proceeds through manual, script-driven workflows in which an analyst selects datasets by hand, authors bespoke Python or R code for each question, and reconciles findings across sources through personal judgment. Each of these acts is undocumented and irreproducible: the rationale for choosing one dataset over another exists only in the analyst's reasoning, and the synthesis step leaves no auditable trace. The workflow is also unscalable, degrading as the surveillance environment fragments into heterogeneous, multi-dataset collections.

General-purpose agentic systems do not resolve this. Lacking domain structure, they possess no mechanism for deciding which datasets bear on a given query and no means of enforcing analytical reproducibility; relevance and rigor are left to the model's parametric knowledge. We introduce a multi-agent system designed to close precisely this gap, supplying the domain scaffolding — explicit dataset indexing, typed analytical plans, and isolated execution — that converts an open-ended language model into a reproducible epidemiological analyst. We call this system SPARQ: a System for Pathogen Analysis, Reasoning, and Querying.

## 2. System Overview

SPARQ decomposes epidemiological analysis into five single-responsibility nodes rather than entrusting it to one agent. This decomposition forecloses a failure mode endemic to single-agent systems: when one model both selects datasets and writes the code that analyzes them, it conflates two decisions that draw on different information — what analysis the question demands, and how to compute it. SPARQ separates these concerns across a *router*, *planner*, *executor*, *aggregator*, connected as a directed graph through which every query passes (Figure []). Each node exposes a formal input/output contract, and the boundary between nodes is itself a checkpoint at which the system's intermediate reasoning becomes inspectable.

At its core, SPARQ is a planner–executor: a planning agent that decides how a query should be answered, and an execution agent that carries that analysis out. Separating these two stages is a recurring pattern in LLM-agent design [Wang et al., 2023; Kim et al., 2024], and it is the pattern SPARQ instantiates for epidemiology. The separation prevents a specific conflation: a single agent deciding how to approach a question while simultaneously writing the code to answer it confuses two decisions that draw on different information — what analysis the question demands, and how to compute it. The planner owns the first. An epidemiological query is rarely a lookup; answering it may require constructing a sequence of analytical steps — exploring a distribution, aggregating to the right unit, testing an association, correlating signals across sources — and which sequence is appropriate depends on domain knowledge about what each dataset measures. The planner works this out and emits a typed, sequence-numbered plan whose steps declare their dependencies — an artifact that also makes the model's reasoning legible, exposing why it chose these steps and these datasets before any code runs, so that the hypothesis behind an analysis can be inspected rather than inferred after the fact. The executor owns the second: because the plan declares dependencies between steps, rather than run them in the order written, it compiles their declared dependencies into an execution topology and dispatches independent steps to sub-agents, each running its code in isolation and returning a typed result. These two nodes are where SPARQ's design decisions live, and the subsections that follow treat each in turn.

Two further nodes surround this core to make it usable and its results consumable; they carry no novel design and are described here only to complete the pipeline (Figure []). A router precedes the planner so that SPARQ can be addressed conversationally rather than handed only well-formed analytical tasks: it classifies each query as analytical or not, passing genuine data questions to the planner and answering out-of-domain queries directly. After execution, an aggregator consolidates the executor's per-step results into a single answer. Because it is a separate node reading results it did not produce, it synthesizes them impartially and can act as a minimal judge — reporting which steps failed or returned little rather than papering over the gaps — and its system instruction fixes the form the final answer must take, enforcing output formatting at one controllable point rather than leaving it to the executor. Every run is then saved in full — query, plan, per-step results, and final answer — to a durable trace that makes the analysis reproducible in the sense Section 1 demanded, alongside a concise answer file for the end user.

## 3. The Planner: Domain-Grounded Decomposition

The planner performs three functions. It decomposes a natural-language query into an ordered analytical workflow — a sequence of exploratory, aggregative, statistical, and correlational steps over the available data; it grounds that workflow in explicit knowledge of the datasets at hand; and it emits the result as a typed plan that both drives the executor and exposes the reasoning behind it. Together these are what let SPARQ approach a complex epidemiological question reliably where its alternatives cannot.

Decomposition is necessary because an epidemiological query worth asking is rarely answerable by a single operation. Answering it means constructing a sequence of analytical steps — exploring a distribution, aggregating to the right unit, testing an association, correlating signals across sources — and choosing those steps, together with the data they run on, is itself an act of epidemiological reasoning.

SPARQ supplies that knowledge explicitly rather than leaving it assumed. A data-summaries artifact, assembled at runtime and injected into the planner's context, describes every available dataset along two axes: its schema — the datatypes it contains — and its meaning — what it represents, how to load it, and the quirks that govern its correct use. The planner composes its workflow against a catalogue it can read, rather than recalling from parametric memory what a molecular-subtyping registry contains or how an outbreak table is coded — knowledge a static statistical model, fixed to one pre-cleaned dataset, never needs, and a general-purpose agent never has. And because this knowledge lives in a versioned file rather than the model's weights or a fixed prompt, an epidemiologist can correct a description, register a new dataset, or record a newly discovered quirk by editing it; the system's understanding of its data evolves without retraining.

The result is emitted as a typed, sequence-numbered plan — an ordered list of steps, each naming the datasets it draws on, giving its rationale, tagged with a task type, and declaring its dependencies on earlier steps. Produced before any code runs, the plan is the model's stated hypothesis for how the question should be answered, and casting it as a structured artifact makes that hypothesis inspectable. Where a manual analyst's reasoning survives only in memory, and a single-pass agent's is buried in the forward pass that produced its answer, SPARQ's plan lets an epidemiologist see why these analytical steps were chosen — which analyses, over which data, in what order — and challenge the approach before any result is trusted. Its declared dependencies, in turn, give the executor the structure it compiles into a parallel execution topology — the subject of the next subsection.

## 4. The Executor: Parallel Execution

The executor runs the plan. Because each step declares which earlier steps it depends on, the executor follows those dependencies rather than a fixed sequence: at each turn it hands every step whose dependencies are already met to its own sub-agent and runs those sub-agents in parallel, before moving on to the next set. Each sub-agent's code is generated by the model and runs in isolation, under a time limit, so a failure in one cannot disturb the others or the system coordinating them.

What the executor must resolve is how the work one sub-agent produces reaches the others that need it. Each sub-agent works on its own, building up the datasets, models, and intermediate results its analysis requires. Before a dependent step begins, the outputs of every step it depends on — both the data they produced and the findings they reported — are combined and handed to its sub-agent, so that everything produced upstream is available to it while nothing from unrelated branches is. Results flow along the dependencies the planner declared, and nowhere else.

## 6. Limitations

Three specific failure modes warrant mention. First, the planner's reasoning is bounded by the completeness of the data-summaries artifact — datasets absent from the manifest are invisible to the system, so the planner's workflow cannot incorporate them. Second, the executor's generated code can produce analytically plausible but statistically incorrect results that pass execution cleanly and return a result; this requires human review of the findings before they are trusted. Third, parallelization is only as sound as the plan's declared dependencies: if a step depends on another but that dependency is not declared, the two steps may be scheduled to run in parallel and race on shared assumptions, producing incorrect results.
